# chat-summarizer

A self-hosted tool that monitors Rocket.Chat and Microsoft Teams for chat threads you care about and uses a **locally-running LLM** (via [Ollama](https://ollama.com)) to generate per-thread summaries. No message content ever leaves your machine.

## What it does

Most chat summarizers sweep entire channels. This tool is different: it tracks only the **specific threads that matter to you** ŌĆö threads you started, threads you're following on the platform, and threads where you've been @mentioned or have replied. For each tracked thread it maintains:

- **`raw.md`** ŌĆö a cumulative, append-only verbatim transcript of every message in the thread, with timestamped section markers each time new messages arrive.
- **`summary.md`** ŌĆö a complete LLM-generated summary, regenerated from scratch whenever new activity appears in the thread.

Output is organized on disk as:

```
summaries/
  <platform>/
    [<team>/]
      <channel>/
        <YYYY-MM-DD>/           ŌåÉ date thread originated
          <thread-id>/
            raw.md
            summary.md
```

Thread state is stored in **PostgreSQL**: which threads are tracked, when they were last active, and whether they need re-summarization. After a configurable idle period (default 90 days), threads are flagged as aged-out. Nothing is deleted automatically ŌĆö the `--prune` command lets you review and confirm deletions interactively.

### How threads are discovered

| Reason | Rocket.Chat | Microsoft Teams |
|---|---|---|
| **started** | You authored the top-level post | You authored the top-level post |
| **following** | Platform-native (`chat.getThreadsList?type=following`) | Not available (no API) |
| **mentioned** | `@yourname` appears anywhere in the thread | Your name in `mentions[]` array or post body |
| **replied** | You posted a reply to the thread | *(planned ŌĆö expensive for Teams)* |

### Supported platforms

- **Rocket.Chat** ŌĆö REST API with Personal Access Token auth
- **Microsoft Teams** ŌĆö Microsoft Graph API with OAuth2 client credentials

### LLM runtime

Summaries are generated by a locally-running [Ollama](https://ollama.com) instance. The default model is **Llama 3.1 8B** (~5 GB VRAM). Mistral 7B (Apache 2.0) is a drop-in alternative. Both run on CPU if you have no GPU, though they'll be slower (30ŌĆō120 seconds per summary vs. 3ŌĆō8 seconds with a GPU). Ollama can run inside Docker Compose alongside the app, or you can point the tool at an Ollama instance already installed on your host machine ŌĆö see [Using a host-installed Ollama](#using-a-host-installed-ollama).

## Quick Start

### Prerequisites

- [Docker Engine](https://docs.docker.com/engine/install/) 25+
- Docker Compose v2 (`docker compose` ŌĆö note: no hyphen)
- An NVIDIA GPU is optional but recommended for fast summarization
- Python 3.14 (preferred) and [uv](https://docs.astral.sh/uv/) ŌĆö only needed for local scripts (`scripts/seed_rocketchat.py`) and running tests; the app itself always runs inside Docker regardless of your local Python version

### Local Python environment (for scripts & tests)

The app runs entirely inside Docker (its image pins its own Python version ŌĆö see `Dockerfile`), so you don't need Python locally just to run `chat-summarizer`. A local environment is only needed for one-off scripts like `scripts/seed_rocketchat.py` and for running the test suite. **Python 3.14 is preferred.**

Install [uv](https://docs.astral.sh/uv/), a fast drop-in replacement for `pip`/`venv`:

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
```

From the repo root, create a `.venv` pinned to Python 3.14 and install dependencies into it:

```bash
uv venv --python 3.14
source .venv/bin/activate
uv pip install -r requirements.txt
```

> **Note:** venvs created by `uv` don't ship a `pip` binary. Always install packages with `uv pip install <package>`, not plain `pip install` ŌĆö with the venv activated, a bare `pip install` silently falls through to your system/pyenv Python and installs *outside* `.venv`, so the package looks installed but `python scripts/seed_rocketchat.py` still fails with `ModuleNotFoundError`.

### 1. Clone and configure

```bash
git clone https://github.com/your-org/chat-summarizer.git
cd chat-summarizer

cp .env.example .env
cp config.yaml.example config.yaml
```

Open `.env` and fill in your credentials:

```bash
# Rocket.Chat ŌĆö base URL + port, and a token generated at: My Account ŌåÆ Security ŌåÆ Personal Access Tokens
RC_URL=https://your-rocketchat.example.com
RC_PORT=3030
RC_USER_ID=your_rocketchat_user_id
RC_AUTH_TOKEN=your_personal_access_token

# Microsoft Teams ŌĆö see DESIGN.md ┬¦7 for Azure app registration steps
TEAMS_TENANT_ID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
TEAMS_CLIENT_ID=yyyyyyyy-yyyy-yyyy-yyyy-yyyyyyyyyyyy
TEAMS_CLIENT_SECRET=your_client_secret_value

# Your personal Azure AD Object ID (not the app's client_id)
# Find it: Azure Portal ŌåÆ Azure AD ŌåÆ Users ŌåÆ your account ŌåÆ Object ID
TEAMS_ME_USER_ID=zzzzzzzz-zzzz-zzzz-zzzz-zzzzzzzzzzzz

# PostgreSQL ŌĆö these values initialize the database on first start
POSTGRES_HOST=postgres
POSTGRES_DB=chat_summarizer
POSTGRES_USER=summarizer
POSTGRES_PASSWORD=change_me_in_production
```

> **Optional ŌĆö custom CA bundle:** if your Rocket.Chat instance uses a self-signed or internal-CA certificate rather than one from a publicly trusted CA, set `RC_CA_BUNDLE` in `.env` to the path of a CA bundle file *inside the app container* for the app to trust when connecting to `RC_URL`. Since it's an in-container path, you'll also need to bind-mount the actual cert file to that path in `docker-compose.yml`'s `app` service ŌĆö e.g. `- ./my-ca.crt:/certs/my-ca.crt:ro` alongside `RC_CA_BUNDLE=/certs/my-ca.crt`. Not needed against a normally CA-signed instance. (The local testing stack below already wires this up automatically for its own self-signed proxy cert ŌĆö no action needed there.)

### 2. Edit `config.yaml`

`config.yaml` is gitignored (like `.env`) ŌĆö it's your personal copy made from `config.yaml.example` above, so local edits never get committed. Open it and make three edits:

**a) Set the channels you want to monitor:**

```yaml
rocketchat:
  enabled: true
  url: "${RC_URL}:${RC_PORT}"   # set RC_URL and RC_PORT in .env, above
  channels:
    - name: "general"
    - name: "engineering"

teams:
  enabled: false   # set to true once Azure credentials are configured
  teams_channels:
    - team_name: "Engineering"
      team_id: "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
      channels:
        - channel_name: "general"
          channel_id: "19:xxxxxxxxxxxxxxxxxx@thread.tacv2"
```

**b) Set your identity** (used to detect threads you started or were mentioned in):

```yaml
me:
  rocketchat:
    user_id: "${RC_USER_ID}"
    username: "yourname"          # plain username, no @
  teams:
    user_id: "${TEAMS_ME_USER_ID}"
    display_name: "Your Full Name"
```

**c) Adjust tracking behavior (optional):**

```yaml
tracking:
  thread_age_out_days: 90        # flag threads idle for this many days
  first_run_lookback_days: 90    # on first run, look back this far for threads you started
```

### 3. Pull the LLM model

This step downloads the model weights into `./ollama-models/` (persisted as a local directory, not a Docker volume):

```bash
docker compose run --rm ollama ollama pull llama3.1:8b
```

For a fully Apache-licensed alternative:

```bash
docker compose run --rm ollama ollama pull mistral:7b
# then set model: "mistral:7b" in config.yaml
```

### 4. Start the database and Ollama

```bash
docker compose up -d postgres ollama
```

Wait about 20 seconds for both services to report healthy:

```bash
docker compose ps
```

### 5. Run

The `app` container runs as your host user rather than a fixed user baked into the image, so files it writes under `./summaries/` and `./logs/` end up owned by you instead of some unrelated container UID. For that to work, export `APP_UID`/`APP_GID` before running ŌĆö **not** `UID`/`GID`; `$UID` is a read-only bash builtin and can't be exported:

```bash
export APP_UID=$(id -u) APP_GID=$(id -g)
docker compose run --rm app
```

If unset, both default to `1000`, which is right for most single-user Linux setups but may not match yours ŌĆö run `id -u` / `id -g` to check. Add the `export` line to your shell profile (`~/.bashrc`, `~/.zshrc`, etc.) so you don't need to repeat it every session; every `docker compose run --rm app` invocation in this README (and in cron, below) needs it set.

On first run, the tool looks back `first_run_lookback_days` for threads you started. Subsequent runs process only activity since the last run (tracked in `state.json`).

Summaries appear in `./summaries/`. Logs go to `./logs/summarizer.log`.

### 6. Schedule with cron

Add a crontab entry to run automatically. Every 30 minutes is a reasonable starting point. Cron runs jobs with a minimal environment ŌĆö it won't have `APP_UID`/`APP_GID` from your shell profile, so set them inline in the crontab entry itself, replacing `1000`/`1000` below with your own `id -u`/`id -g` output:

```cron
*/30 * * * * export APP_UID=1000 APP_GID=1000; cd /path/to/chat-summarizer && docker compose run --rm app >> /path/to/chat-summarizer/logs/cron.log 2>&1
```

> **Note:** Never pass `--prune` to a cron job. The prune command requires interactive TTY input.

## Pruning aged-out threads

After `thread_age_out_days` days of inactivity, threads are flagged `aged_out` in Postgres but nothing is deleted. To review and clean them up:

```bash
docker compose run -it --rm app --prune
```

You'll be shown each aged-out thread one at a time and asked to confirm deletion. Answering `y` deletes both the Postgres records and the `summaries/` directory for that thread. Answering anything else skips it.

## Using a host-installed Ollama

If Ollama is already running on your machine (installed natively, not via Docker), you can skip the `ollama` service entirely and point the app at your host's instance. This is often preferable ŌĆö you get faster startup, you can switch models freely using your existing `ollama` CLI, and you avoid running two Ollama processes.

### Step 1 ŌĆö Make sure the model is pulled on the host

```bash
ollama pull llama3.1:8b
```

### Step 2 ŌĆö Update the Ollama URL in `config.yaml`

The Docker app container cannot reach `localhost` on the host directly. Use the special hostname that Docker provides for this purpose:

```yaml
ollama:
  url: "http://host.docker.internal:11434"   # macOS and Windows (Docker Desktop)
  model: "llama3.1:8b"
```

On **Linux** with Docker Engine (not Docker Desktop), `host.docker.internal` is not available by default. Use either of these approaches:

**Option A ŌĆö add an extra host entry to `docker-compose.yml`:**

```yaml
services:
  app:
    extra_hosts:
      - "host.docker.internal:host-gateway"
```

Then use `http://host.docker.internal:11434` in `config.yaml` as above.

**Option B ŌĆö use the docker bridge gateway IP directly** (less portable):

```bash
# Find the gateway IP
ip route show default | awk '/default/ {print $3}'
# Typically 172.17.0.1
```

```yaml
ollama:
  url: "http://172.17.0.1:11434"
```

### Step 3 ŌĆö Start only the database

```bash
docker compose up -d postgres
```

Skip the `ollama` service entirely. You can also remove or comment it out of `docker-compose.yml` to avoid accidentally starting it.

### Step 4 ŌĆö Run as usual

Remember `APP_UID`/`APP_GID` still need to be exported (see [Run](#5-run) above) if they aren't already:

```bash
docker compose run --rm app
```

The app will connect to your host's Ollama on port 11434.

## CPU-only setup

If you have no NVIDIA GPU, remove the `deploy` block from the `ollama` service in `docker-compose.yml`:

```yaml
ollama:
  image: ollama/ollama:latest
  # remove the 'deploy:' section entirely
```

Then use a smaller model to keep inference times reasonable:

```bash
docker compose run --rm ollama ollama pull phi3:mini
# set model: "phi3:mini" in config.yaml
```

## Testing with a local Rocket.Chat instance

`docker-compose.yml` includes a complete Rocket.Chat stack (Rocket.Chat + MongoDB + a TLS-terminating reverse proxy) under a `testing` Docker Compose profile. It starts independently of the normal services and gives you a throwaway server to develop and test against without touching a production instance.

Rocket.Chat itself has no host port ŌĆö it's reached exclusively through `rocketchat-proxy`, an nginx reverse proxy that terminates HTTPS with a self-signed certificate (generated on first start by `rocketchat-proxy-cert-init`) and forwards to `rocketchat:3000` inside the Docker network.

### Start the test stack

```bash
# Start Postgres, Ollama, AND the local Rocket.Chat instance
docker compose --profile testing up -d

# Or start only the Rocket.Chat stack (if you already have Postgres/Ollama running)
docker compose --profile testing up -d mongo mongo-init-replica rocketchat rocketchat-proxy-cert-init rocketchat-proxy
```

Rocket.Chat takes 30ŌĆō60 seconds to become ready. Watch its log output until you see `SERVER RUNNING` before proceeding:

```bash
docker compose logs -f rocketchat
```

### Complete the setup wizard

Open **https://localhost:3030** in your browser. Your browser will warn about the self-signed certificate (`rocketchat-proxy` generated its own, not one from a trusted CA) ŌĆö this is expected for a local test instance; click through the warning (usually "Advanced" ŌåÆ "Proceed anyway"). Rocket.Chat shows a one-time setup wizard on first start. Work through it to create the admin account ŌĆö use whatever credentials you like for testing, for example:

| Field | Example value |
|---|---|
| Admin name | Admin |
| Username | `admin` |
| Email | `admin@localhost` |
| Password | `changeme` |

> **Tip:** You can skip the wizard entirely ŌĆö the `ADMIN_*` environment variables on the `rocketchat` service in `docker-compose.yml` are already active by default and auto-provision the admin account on first boot.

Finish the wizard and log in as admin. On the organization and server-info screens, choose any values and click through ŌĆö they don't affect local testing.

### Create a test user (the "me" account)

The summarizer tracks threads belonging to a specific user. Create a dedicated test user that will act as "you":

1. In the Rocket.Chat admin panel, go to **Administration ŌåÆ Users ŌåÆ New User**.
2. Fill in a name, username, and email ŌĆö for example, username `testuser`.
3. Set a password, uncheck "Require password change", and click **Save**.

### Create a Personal Access Token

The summarizer authenticates with Rocket.Chat using a Personal Access Token tied to the test user, not the admin account.

1. Log out of the admin account and log in as `testuser`.
2. Go to **My Account ŌåÆ Security ŌåÆ Personal Access Tokens**.
3. Give the token a name (e.g. `chat-summarizer`) and click **Add**.
4. Copy the **User ID** and **Token** ŌĆö you will not be able to see the token again.

### Configure the summarizer

Update `.env` with the test credentials. `RC_URL`/`RC_PORT` should point at Rocket.Chat's internal Docker network hostname, not the HTTPS proxy ŌĆö the app container reaches Rocket.Chat directly at `http://rocketchat:3000`, bypassing `rocketchat-proxy` entirely (that proxy exists only for host-browser access):

```bash
RC_URL=http://rocketchat
RC_PORT=3000
RC_USER_ID=<User ID copied above>
RC_AUTH_TOKEN=<Token copied above>
```

`config.yaml` already picks this up automatically via `${RC_URL}:${RC_PORT}`, no edit needed there:

```yaml
rocketchat:
  enabled: true
  url: "${RC_URL}:${RC_PORT}"
  user_id: "${RC_USER_ID}"
  auth_token: "${RC_AUTH_TOKEN}"
  channels:
    - name: "general"

me:
  rocketchat:
    user_id: "${RC_USER_ID}"
    username: "testuser"         # must match the username you created above
```

Also set `POSTGRES_HOST=postgres` in `.env` if it isn't already.

### Seed some test threads

The fastest way to get realistic test data is the seeding script, which creates a handful of fictional users, five channels (`general`, `engineering`, `incident-response`, `product-launch`, `random`), and a set of multi-user threaded conversations ŌĆö including threads started by, replying to, mentioning, and followed by `testuser` ŌĆö so every branch of the thread-eligibility logic has something to find:

```bash
# requires the local Python environment set up above (uv + Python 3.14)
uv pip install -r requirements.txt   # httpx and tenacity are already listed there
python3 scripts/seed_rocketchat.py
```

It logs in as `admin`/`admin` (see `ADMIN_*` in `docker-compose.yml`) to provision everything, so run it after the setup wizard has been skipped/completed. Safe to re-run ŌĆö users and channels are reused, though each run adds another copy of the conversations.

Alternatively, seed data manually. Log back into Rocket.Chat as `testuser` and create content for the summarizer to find:

1. Go to the **#general** channel (created by default).
2. Post a message to start a new thread ŌĆö hover over it, click the **thread** icon, and add a few replies.
3. Optionally log in as a second browser session as `admin` and reply to the same thread to simulate other participants.

### Run the summarizer against the local instance

Remember `APP_UID`/`APP_GID` still need to be exported (see [Run](#5-run) above) if they aren't already:

```bash
docker compose --profile testing run --rm app
```

Summaries appear in `./summaries/rocketchat/general/`. You can re-run as many times as you like; only threads with new activity since the last run will be re-summarized.

### Reset the test environment

To wipe all Rocket.Chat data and start fresh:

```bash
docker compose --profile testing down -v
```

The `-v` flag removes the `mongo-data` and `rocketchat-proxy-certs` volumes (the self-signed cert is regenerated automatically on next start, so no extra step needed there). Postgres data (`postgres-data`) is unaffected ŌĆö your summarizer state stays intact unless you explicitly remove that volume too.
